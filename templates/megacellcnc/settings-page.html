Celery Beat – Geräte-Status

Hintergrundaufgabe check_all_devices prüft regelmässig, ob deine Testgeräte erreichbar sind (wie in den Celery-Logs). Das Intervall gilt für den django-celery-beat Eintrag in der Datenbank (Scheduler: DatabaseScheduler).

Bereich 5–3600 Sekunden. Kleinere Werte = häufigere Logs und Netzlast.

Änderungen werden vom Beat-Prozess meist innerhalb weniger Sekunden übernommen. Falls nichts passiert: docker-compose restart celerybeat (oder den Celery-Beat-Container neu starten).

Database Backup & Restore
Backup erstellen

Erstelle ein vollständiges Backup der Datenbank. Download als .json.gz – eine Kopie wird zusätzlich unter „Backups auf dem Server“ abgelegt.

Backup wiederherstellen (Datei vom PC)

Datei von deinem Rechner wählen (.json.gz oder .json). Vor dem Einspielen wird ein Sicherheits-Backup erzeugt und in der Liste unten gespeichert (sicherheit_vor_restore_…).

Datenbank-Schema (Migrationen)

Empfohlen (Docker): Beim Start des Web-Containers wird python manage.py migrate --noinput automatisch ausgeführt, bevor Gunicorn startet. Neues Image ziehen und Container neu starten reicht – keine manuelle Migration nötig.

Lokal ohne Docker: Virtuelle Umgebung aktivieren, pip install -r requirements.txt, dann python manage.py migrate. Der Fehler „No module named django“ bedeutet: Abhängigkeiten fehlen oder venv nicht aktiv.
